## Section 2: Engine Changes

### Fleet tab scope

Fleet notifications cover **harvest and transport only**. Attack outcomes are exclusively in `combatLog` (Combat tab). No double-counting of unread badge.

### Notification creation ŌĆö `FleetEngine.ts`

**Timing:** notifications are created at **target arrival** (`at_target` transition), not at `completed`. Reason: `resolveTransportAtTarget()` mutates `mission.cargo` to the undelivered remainder before `completed` is set, so cargo data is lost by then.

**Idempotency guard** (prevents duplicates across live tick, offline catch-up, admin fast-forward):

```ts
const alreadyNotified = state.fleetNotifications.some(n => n.missionId === mission.id);
if (!alreadyNotified) {
  state.fleetNotifications.push({ id: `${mission.id}-notif`, missionId: mission.id, ... });
}
```

**Failure/empty outcomes:**

| Scenario | `loot` value | `targetName` |
|---|---|---|
| Successful harvest | actual metal/crystal collected | NPC colony name |
| Debris field already cleared | `{ metal: 0, crystal: 0, deuterium: 0 }` | debris coords label |
| Successful transport | cargo delivered | destination planet name |
| Transport target planet gone | `{ metal: 0, crystal: 0, deuterium: 0 }` | coords label (planet gone) |

UI renders zero-loot notifications with a "nothing delivered/collected" hint instead of blank resource rows.

### `EspionageEngine.ts`

Store `detectionChance` on `EspionageReport` at generation time (already computed transiently, now persisted).

## Section 3: UI ŌĆö MessagesPanel

### Nav wiring (three files)

### Panel layout

**File:** `src/panels/MessagesPanel.tsx`

Three tabs: **Combat | Espionage | Fleet**

Each tab:
- Newest-first list (`[...arr].sort((a, b) => b.timestamp - a.timestamp)` ŌĆö never mutate state arrays)
- Compact row; click to expand inline ŌåÆ auto-marks as read
- "Mark all read" button per tab
- Delete button (├Ś) per message
- Empty state: "No [combat / espionage / fleet] messages"

### Combat expanded view

```

  Attacker losses: 12├Ś Light Fighter
```

Includes `defencesRebuilt` from `CombatResult` ŌĆö without it, "Defender losses" is misleading since some defences regenerate.

### Espionage expanded view

Shows all fields present on `EspionageReport`. Tech-gated fields (`fleet`, `defences`, `buildings`) show **"Unknown (insufficient tech level)"** when absent ŌĆö not blank. Shows `detectionChance` as counter-espionage %.

### Fleet expanded view

Uses `targetName` from the notification snapshot ŌĆö does not do a live planet lookup. For zero-loot notifications, shows a hint ("Debris field was already cleared" / "Destination planet no longer exists").

### `refreshArrayReferences()` (`src/hooks/useGameEngine.ts`)

Add `fleetNotifications` clone alongside existing array clones:

```ts

```

### Dangling reference policy

`FleetMission.combatResultId` / `espionageReportId` are **not** cleared when messages are deleted. All UI code that dereferences these IDs must handle `undefined` / missing entry gracefully. No cascading mission update required.

### Admin surface updates

- `adminRegenerateGalaxy()` ŌĆö clears `fleetNotifications` alongside `combatLog` + `espionageReports`
- `adminMarkAllRead()` ŌĆö removed; callers use the three per-tab actions
