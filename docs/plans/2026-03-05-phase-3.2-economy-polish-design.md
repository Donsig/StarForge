# Phase 3.2 — Economy Polish: Design Document

**Date:** 2026-03-05
**Status:** Approved — revised after Codex review
**State version:** v9 → v10

---

## Overview

Four features grouped into three implementation clusters. Goal: OGame-accurate economy mechanics — colony gating via Astrophysics research, slot-based planet properties, solar satellites as a scalable energy source (for both players and NPCs), and inter-planet resource transport.

---

## Cluster 1 — Planet Data (Astrophysics + Slot Properties)

### Model changes

Add to `PlanetState` (`src/models/GameState.ts`):
```ts
temperature: number;   // max surface temp in °C — affects deuterium production + solar satellite output
fieldCount: number;    // building slot capacity (enforced in a future phase)
```

### Slot → temperature/field ranges

| Slots | Temp range (°C) | Field range |
|-------|----------------|-------------|
| 1–3   | 200–400        | 40–70       |
| 4–6   | 100–200        | 90–130      |
| 7–9   | 0–100          | 140–180     |
| 10–12 | −50–0          | 120–160     |
| 13–15 | −100 to −50    | 80–120      |

### Seeded assignment

`mulberry32(galaxySeed ^ (system * 1000 + slot * 17))` — picks a value uniformly within the slot's range. Applied at colonization time and retroactively in migration.

**Recolonization:** when a player abandons and recolonizes the same slot, stats are rerolled using `mulberry32(Date.now() ^ (system * 1000 + slot * 17))` — OGame-like variance, giving fresh values each time.

Homeworld (slot 8, 7–9 range) gets seeded values via migration — no special-casing needed.

### Deuterium formula

Already uses temperature (`-0.002 * temp + 1.28`). After this change, reads `planet.temperature` instead of a hardcoded value. No formula change — only the data source changes.

### Astrophysics research (`src/data/research.ts`)

New research entry:
```ts
{
  id: 'astrophysicsTechnology',
  name: 'Astrophysics',
  baseCost: { metal: 4000, crystal: 8000, deuterium: 4000 },
  costMultiplier: 1.75,
  prerequisites: { espionageTechnology: 4, impulseDrive: 3 }
}
```

### Colony cap enforcement (`src/engine/GalaxyEngine.ts`)

`canColonize()` adds check:
```
maxColonies = floor(astrophysicsLevel / 2) + 1
currentColonies = state.planets.length - 1   // exclude homeworld
return currentColonies < maxColonies
```

Level 1 → 1 colony, level 3 → 2 colonies, level 5 → 3 colonies, etc.

## Cluster 2 — Solar Satellites

### Ship definition (`src/data/ships.ts`)

New ship:
```ts
{
  id: 'solarSatellite',
  name: 'Solar Satellite',
  cost: { metal: 0, crystal: 2000, deuterium: 500 },
  // no drive — immobile, cannot be dispatched
  // combat stats: participates as defender, destroyed in combat
}
```

- No prerequisites (buildable once shipyard exists)
- Excluded from fleet dispatch form and all mission types
- Visible in Shipyard (build) and Overview (energy contribution)

### Player energy integration (`src/engine/ResourceEngine.ts`)

```ts
energyFromSatellites = planet.ships.solarSatellite * Math.floor((planet.temperature + 140) / 6)
```

Added to energy produced total alongside solar plant output. Feeds into existing energy ratio that scales mine output.

### NPC model changes (`src/models/Galaxy.ts`)

Add to `NPCColony`:
```ts
temperature: number;  // assigned at generation, seeded from coordinates
```

### NPC energy balance (`src/engine/GalaxyEngine.ts` — `getNPCResources`)

```
energyProduced = solarPlant formula + (solarSatellite count * floor((temperature + 140) / 6))
energyConsumed = sum of mine energy consumption
energyRatio    = min(1, energyProduced / energyConsumed)
// mine output scaled by energyRatio
```

### NPC generation

- `temperature` assigned from slot using same seeded RNG as player planets
- Initial satellite counts scaled by tier (enough to prevent heavy energy penalties for miners; other specialties get modest counts)

### NPC upgrade system — satellite building

- **`miner`** specialty: builds satellites on `tick % 3 === 2` (same slot as deuterium, alternating)
- **`balanced`** specialty: builds a small batch every 4th tick

After each satellite upgrade, both `baseShips.solarSatellite` and `currentShips.solarSatellite` update (same as other ship upgrades). After a raid, destroyed satellites reduce current count and the energy deficit persists until rebuild over 48 hours.

## Cluster 3 — Inter-Planet Transport

### Mission type extension (`src/models/Fleet.ts`)

Add `'transport'` to `MissionType`. Lifecycle:
```
outbound → at_target (deposit cargo up to target storage cap, surplus stays in mission.cargo) → returning → completed (ships return with any undelivered surplus)
```

### FleetPanel dispatch form

**When mission type is `'transport'`:**

- If initiated from galaxy view (player-owned planet slot clicked): target is pre-filled (same pattern as attack flow)
- If initiated from Fleet panel directly: inline **"Send To" dropdown** appears listing all player-owned planets except the source planet
- Below target: cargo manifest with three number inputs (metal / crystal / deuterium), each capped at `min(available on source, remaining cargo capacity)`
- "Max" button fills all three to capacity proportionally (or fills metal first, then crystal, then deuterium — OGame default order)

### GalaxyPanel changes (`src/panels/GalaxyPanel.tsx`)

Player-owned planet slots show a **"Transport"** button. Clicking it navigates to the Fleet panel with target coordinates and mission type pre-filled.

### FleetEngine changes (`src/engine/FleetEngine.ts`)

**`dispatch` validation for transport:**
- Target must be a player-owned `PlanetState`
- Fleet must contain at least one ship with `cargoCapacity > 0`
- `sum(cargo) ≤ fleetCargoCapacity(ships)`
- Resources deducted from source planet immediately on dispatch

**`at_target` handler:**
- Find target `PlanetState` by coordinates
- Deposit as much of `mission.cargo` as fits within target storage caps; update `mission.cargo` to the undelivered remainder
- Set `status = 'returning'`, `returnTime = now + travelSeconds * 1000`

**`returning → completed`:**
- Ships re-added to source planet fleet
- Any remaining `mission.cargo` (undelivered surplus) deposited back to source planet (capped at source storage)

### Fleet slot usage

Transport consumes one fleet slot. No change to slot counting.

## Open questions / implementation notes

- **Cargo fill order for "Max" button**: fill metal first, then crystal, then deuterium (OGame default). Document in code.
- **Transport fuel cost**: deuterium consumed for the round trip (outbound + return) at dispatch time, same as attack missions. Deducted from source planet alongside the cargo resources.
- **Transport cancel**: inherits existing mission cancel behavior — if outbound, ships and cargo return immediately; if returning, no action needed (already heading home with any surplus).
- **`solarSatellite` in combat**: satellites are in `currentShips` and treated as regular defender units. No special rapid-fire rules needed for satellites (they have no attack).
- **Astrophysics prerequisite gate**: players who already have > 1 colony when this migration lands keep their colonies — migration does NOT remove colonies, it only prevents new ones until astrophysics is researched.
- **NPC satellite counts at generation**: suggested formula — `floor(tier * 1.5)` satellites base, doubled for miner specialty. Tune in implementation.
- **Field count enforcement**: `fieldCount` is stored but not enforced yet (no build slot cap UI). Noted for Phase 3.4.
